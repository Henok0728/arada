Module 1 — Connectivity Resilience

4-Layer Fallback Stack

L1
Redis Availability Cache (90s TTL)
All availability data served from cache. Connectivity blips are invisible to users.
L2
Celery Webhook Queue (Exponential Backoff)
Failed notifications queued persistently. Auto-delivers when hotel reconnects. Max retry: ~68 min.
L3
HMAC Offline Handshake Code
Cryptographically verifiable 6-char code. PWA validates without calling servers.
L4
SMS as Transport Layer (Always-On)
Every referral dispatched via SMS (AfricasTalking). Guaranteed delivery without data.

Module 7 — Digital Tourist Corridor

Expansion Sequence

Stage 1
Hotel ↔ Hotel (Now)
Pure referral network. The foundational trust layer. Months 1–12.
Stage 2
Hotel → Transfer (Month 12–18)
Jiren Ride + Ride Ethiopia integration. Transfer offered at referral confirmation. 5% corridor commission.
Stage 3
Hotel → Tour Guide (Month 18–24)
ETO-certified guide database. Subscription model (not commission) for guide quality alignment.
Stage 4
Tourist Corridor API (Month 24+)
Unified public API: hotel + guide + transport packages. Lodge-Link becomes Ethiopian tourism infrastructure.
Module 8 — Phase 1 MVP Checklist

Critical Launch Tasks

  • ▶ PostgreSQL multi-tenancy: shared platform schema + private hotel schemas
  • ▶ FastAPI async fanout with 3s hard timeout per hotel check
  • ▶ API key SHA-256 hashing — plaintext keys never stored
  • ▶ Row-Level Security (RLS) activated on all private tables
  • ▶ HMAC-signed offline handshake code generation
  • ▶ JWT auth: 15-min access token + 7-day refresh
  • ▶ Celery + Redis webhook queue with exponential backoff
  • ▶ SMS gateway (AfricasTalking) for every referral notification
  • ▶ PWA offline cache: last 50 pending referrals (IndexedDB)
  • ▶ Amharic + English UI (Noto Sans Ethiopic)
  • ▶ Emergency referral mode: 3-tap UI (<10 seconds)
  • ▶ KYC document flow + manual admin review queue
